# 세력 관계 게임화 플랜

## 목적

세력도 페이지를 단순 우호도 현황판에서 세력 관계를 운영하는 ERP 웹게임 화면으로 확장한다. 사용자는 각 세력의 현재 관계 상태, 다음 단계 혜택, 접선 가능 행동, 의뢰 후보를 확인하고, 이후 단계에서 크레딧 기부나 퀘스트 수행으로 관계를 변화시킬 수 있어야 한다.

이 문서는 기능 설계 기준이다. 로어 확정 기록과 게임 시스템 제안은 분리한다.

## 로어 기준

현재 확인된 세력 축은 다음과 같다.

| code | 표시명 | 분류 | 게임화 방향 |
|---|---|---|---|
| `COUNCIL` | 세계이사회 | 외부 권력 블록 | 예산, 승인, 정치적 보호, 고위 정보 |
| `MILITARY` | 군부 | 외부 권력 블록 | 무장 지원, 합동 작전, 선제 격리 의뢰 |
| `CIVIL` | 시민사회 | 외부 권력 블록 | 여론, 민간 제보, 피해자 보호, 시장 루트 |
| `WHITE_ROSE` | 백장미단 | 시민사회 하위 세력 | 공개, 피해자 권리, 제보망, 은폐 리스크 |
| `SPACE_ZERO` | 스페이스 제로 | 시민사회 하위 세력 | 기술 자본, 특수 장비, 샘플 거래, 연구 의뢰 |
| `HOSTILE` | 적대세력 | 작전상 적대 분류 | 우호보다 위협 통제, 침투, 단서 확보 |
| `GOLDEN_DAWN` | 황금여명회 | 적대 하위 세력 | 컬트 단서, 의식 방해, 은신처 추적 |
| `AHNENERBE` | 아넨에르베 "광명회" | 적대 하위 세력 | 실험 기록, 연구 시설, 적대 개체 약점 |

적대세력은 일반 세력과 같은 UI 골격을 쓰되, 문구는 `우호도`보다 `위협 통제`, `침투도`, `정보 접근도`에 가깝게 해석한다.

## 관계 단계

1차 MVP는 숫자 계산만 사용하고 자동 보상 지급은 하지 않는다.

| 범위 | 일반 세력 라벨 | 적대세력 라벨 | 의미 |
|---:|---|---|---|
| -10..-7 | 적대 | 통제 실패 | 접촉 위험. 패널티/감시 이벤트 후보 |
| -6..-3 | 불신 | 위협 고조 | 제한 접촉. 비용 증가 후보 |
| -2..2 | 중립 | 추적 중 | 기본 브리핑과 제한 행동만 가능 |
| 3..5 | 협조 | 단서 확보 | 기본 접선/의뢰 후보 개방 |
| 6..8 | 우호 | 침투 성공 | 특수 혜택/고급 정보 후보 |
| 9..10 | 핵심 후원 | 핵심 약점 확보 | 고급 보상/결정적 작전 카드 후보 |

## UX 구조

### 세력도 메인

- 전체 관계도와 현재 수치를 유지한다.
- 선택한 세력의 우측 브리핑에 다음 정보를 추가한다.
  - 현재 관계 단계
  - 다음 단계까지 필요한 점수
  - 현재 해금된 혜택 요약
  - 세력 상세/접선 화면 진입 버튼

### 세력 상세 페이지

예상 경로: `/erp/factions/[code]`

구성:

1. 세력 브리핑
   - 로고 또는 식별 sigil
   - 세력명, 분류, doctrine, summary
   - 현재 우호도와 관계 단계

2. 혜택 트랙
   - 단계별 혜택 카드
   - 현재 달성 단계 강조
   - 다음 단계 보상 preview

3. 접선 콘솔
   - 대표 접선 문구
   - 인물 사진 슬롯은 1차 MVP에서 placeholder 없이 텍스트/sigil만 사용
   - 선택지 버튼은 1차 MVP에서 미리보기만 제공

4. 의뢰 후보
   - 로어 기반 quest seed 카드
   - `제안됨` 상태의 정적 카드로 시작
   - 실제 수락/완료/보상 지급은 2차 이후

## 1차 MVP 범위

이번 1차 구현에서 한다.

- 세력도 데이터 생성 로직을 상세 페이지에서 재사용 가능하게 분리한다.
- `/erp/factions/[code]` 상세 페이지를 추가한다.
- 상세 페이지에 관계 단계, 혜택 트랙, 접선 콘솔, 의뢰 후보 정적 UI를 추가한다.
- 세력도 우측 브리핑에서 상세 페이지로 이동하는 버튼을 추가한다.

이번 1차 구현에서 하지 않는다.

- 크레딧 차감
- 우호도 자동 변경
- 퀘스트 수락/완료 DB
- 아이템/상점/인벤토리 보상 지급
- 세력 간 반작용 자동화
- 개인별 평판

## 2차 이후 후보

### 관계 액션

- 크레딧 기부
- 정보 제공
- 작전 협조
- 샘플/자료 전달
- 의뢰 수락
- 거래 거절

### DB 후보

- `faction_contact_actions`: 세력별 접선 선택지
- `faction_quests`: 세력 의뢰와 상태
- `faction_relation_logs`: 관계 변화 로그
- `faction_benefit_rules`: 우호도 단계별 혜택 규칙
- `character_faction_standing`: 개인별 평판, 후순위

### 경제 mutation 원칙

크레딧, 인벤토리, 상점, 주식 등 경제 영향은 GM 승인형 mutation으로 분리한다. UI에서 먼저 preview를 보여주고, 실제 반영은 별도 API와 로그 검증을 붙인 뒤 진행한다.

## 세력별 1차 콘텐츠 방향

| code | 접선 톤 | 1차 혜택 후보 | 1차 의뢰 후보 |
|---|---|---|---|
| `COUNCIL` | 고위 의결 라인 | 예산 승인, 보고서 우선 검토 | 민감 보고서 제출 |
| `MILITARY` | 합동 작전 라인 | 보급품 우선권, 군사 정보 | 위협 개체 선제 격리 |
| `CIVIL` | 민간 조율 라인 | 제보망, 여론 완충 | 피해자 보호 조치 |
| `WHITE_ROSE` | 급진 시민 접촉 | 민간 증언, 피해자 연결 | 은폐 기록 검증 |
| `SPACE_ZERO` | 기술 계약 라인 | 장비 할인, 실험 장치 제안 | 격리 장비 시제품 테스트 |
| `GOLDEN_DAWN` | 침투/추적 라인 | 의식 단서, 은신처 흔적 | 컬트 의식 방해 |
| `AHNENERBE` | 적대 연구 추적 | 실험 기록, 개체 약점 | 연구 시설 자료 회수 |

## 검증 기준

- 인증 없는 접근은 기존 ERP와 동일하게 로그인으로 리다이렉트된다.
- `/erp/factions`에서 기존 관계도와 우호도 수정 기능이 유지된다.
- `/erp/factions/[code]`에서 존재하는 code는 상세 페이지가 렌더된다.
- 없는 code는 `/erp/factions`로 되돌리거나 notFound 처리한다.
- 타입체크와 대상 lint가 통과한다.
- 경제/DB mutation은 1차 MVP에서 발생하지 않는다.
