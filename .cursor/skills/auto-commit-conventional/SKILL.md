# Auto Commit Conventional

## 목적

사용자가 자동 커밋을 요청하면 변경사항을 점검하고, 필요한 파일을 스테이징한 뒤, Conventional Commit 형식으로 커밋을 생성한다.

## 트리거

다음과 같은 요청이 있으면 이 스킬을 우선 적용한다.

- "자동 커밋해줘"
- "스테이징하고 커밋해줘"
- "컨벤셔널 커밋 메시지로 커밋해줘"
- "지금 변경사항 커밋해줘"

## 기본 동작

1. 현재 상태 확인
   - `git status --short`
   - `git diff --staged`
   - `git diff`
   - `git log -5 --oneline`
2. 커밋 전략 선택(필수)
   - 먼저 사용자에게 아래 중 하나를 확인한다.
   - `단일 커밋`: 이번 작업 전체를 하나의 커밋으로 묶음
   - `분할 커밋`: 작업 성격이 같은 단위로 나눠 여러 커밋 생성
3. 커밋 대상 파일 선별
   - 사용자가 범위를 지정하면 해당 범위만 포함
   - 지정이 없으면 현재 작업과 관련된 파일 포함
4. 기본 제외 규칙 적용
   - 제외: `.env`, `.env.*`, `*.pem`, `*.key`, `credentials.json`, `secrets.*`
5. 스테이징 및 커밋 실행
   - 단일 커밋: 관련 파일을 한 번에 `git add` 후 1회 커밋
   - 분할 커밋: 성격별 파일 묶음마다 `git add` + 커밋 반복
   - 분할 기준 예시
     - 기능 추가(`feat`) / 버그 수정(`fix`) / 문서(`docs`) / 설정(`chore`, `build`, `ci`)를 분리
     - API 변경과 UI 변경이 분리 가능하면 별도 커밋
6. 커밋 메시지 생성 (Conventional Commits, 한글 강제)
   - 타입 후보: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `build`, `ci`
   - 형식: `<type>(<scope>): <subject>`
   - 제목(`subject`)은 반드시 한글로 작성
   - 본문 bullet도 반드시 한글로 작성
   - 본문은 "무엇을 왜 바꿨는지"를 1-3개 bullet로 요약
7. 커밋 실행
   - 각 커밋마다 HEREDOC으로 메시지 전달
   - 예시:

```bash
git commit -m "$(cat <<'EOF'
feat(api): 문의 및 지원 폼 디스코드 웹훅 전송 추가

- 서버 API 라우트에서 폼 데이터를 검증 후 웹훅으로 전달하도록 변경
- 폼 제출 로직을 공통 함수로 정리해 중복 코드를 제거
- 배포 환경변수 설정 문서를 추가해 운영 설정 실수를 줄임
EOF
)"
```

8. 검증
   - `git status`로 커밋 완료 여부 확인
   - 결과를 사용자에게 간단히 보고

## 메시지 규칙

- 커밋 제목과 본문은 무조건 한글
- subject는 명령형 현재 시제, 72자 내외
- 불필요한 접두어 금지 (예: "update stuff")
- scope는 가능하면 포함 (`api`, `ui`, `auth`, `build` 등)
- 변경 목적이 드러나는 문장 사용

## 분할 커밋 템플릿

분할 커밋 선택 시 아래 템플릿을 기본으로 사용한다.

- `feat`: `feat(<scope>): <기능 추가 내용>`
- `fix`: `fix(<scope>): <버그 수정 내용>`
- `refactor`: `refactor(<scope>): <구조 개선 내용>`
- `docs`: `docs(<scope>): <문서 변경 내용>`
- `test`: `test(<scope>): <테스트 추가/수정 내용>`
- `chore`: `chore(<scope>): <설정/잡무 변경 내용>`
- `build`: `build(<scope>): <빌드 설정 변경 내용>`
- `ci`: `ci(<scope>): <CI 파이프라인 변경 내용>`

본문 bullet 템플릿(한글 고정):

- `- 무엇을 변경했는지`
- `- 왜 이 변경이 필요한지`
- `- 영향 범위 또는 후속 작업`

예시:

```text
feat(api): 지원서 접수 API 유효성 검증 추가

- 필수 입력 누락 시 400 응답을 반환하도록 서버 검증 로직을 추가
- 클라이언트 우회 요청에서도 동일한 검증을 보장하기 위해 서버 단에서 처리
- /apply 제출 플로우와 오류 메시지 표시 동작에 영향
```

## 안전 규칙 (보통)

- 비밀정보 파일은 기본 제외
- 파괴적 git 명령 금지 (`reset --hard`, `checkout --` 등)
- 원격 푸시는 사용자가 명시적으로 요청한 경우에만 수행
- 커밋 실패(훅/충돌) 시 원인 해결 후 새 커밋 시도

## 사용자 응답 형식

커밋 후 아래 정보를 짧게 전달한다.

- 커밋 해시(짧은 SHA)
- 커밋 제목 1줄
- 포함된 파일 수와 핵심 변경 요약 1-3개
