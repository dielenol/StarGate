# TRPG 일정 참여 체크 디스코드 봇 개발 플랜

## 1. 목표

TRPG 세션 운영을 위해, 디스코드 내에서 참여 가능 여부를 수집하고 마감 시 최종 결과를 집계하는 봇을 만든다.

핵심 목표는 다음과 같다.

- 관리자가 세션을 생성할 수 있어야 한다.
- 참여자들이 쉽게 참석 / 불참 / 미정 또는 날짜별 가능 여부를 표시할 수 있어야 한다.
- 마감 시점에 최종 결과를 자동 집계할 수 있어야 한다.
- 최종 결과는 텍스트 중심으로 제공하되, 이후 캘린더형 시각화로 확장 가능해야 한다.
- 처음부터 모든 기능을 넣기보다, MVP를 먼저 만들고 점진적으로 확장한다.

---

## 2. 제품 방향

초기 단계에서는 **단일 날짜 참석 체크형**을 우선 구현한다.

이유:

- 구현 난이도가 낮다.
- 디스코드 메시지와 버튼만으로도 UX를 충분히 만들 수 있다.
- 일정 조율용 캘린더 기능은 이후 다중 날짜 후보형으로 자연스럽게 확장할 수 있다.

즉, 개발 순서는 아래와 같다.

1. 단일 날짜 세션 생성
2. 참석 / 불참 / 미정 응답 수집
3. 마감 및 최종 집계
4. 무응답자 추적
5. 리마인드 기능 추가
6. 다중 날짜 후보형으로 확장
7. 캘린더형 결과 시각화 추가

## 3. 개발 범위

### 3-1. 1차 MVP 범위

- 관리자 슬래시 명령어로 세션 생성
- 특정 채널에 세션 공지 메시지 생성
- 참여 버튼 제공
  - 참석
  - 불참
  - 미정
- 사용자 응답 저장
- 사용자가 응답을 변경할 수 있음
- 마감 시 자동 종료
- 최종 결과 텍스트 출력
- 대상 역할 기준 무응답자 추출

### 3-2. 2차 확장 범위

- 마감 전 리마인드 발송
- 관리자용 수정 명령어
  - 마감 연장
  - 세션 취소
  - 강제 마감
  - 현재 집계 확인
- 결과 카드 이미지 출력
- 운영 로그 저장

### 3-3. 3차 확장 범위

- 다중 날짜 후보 세션 생성
- 날짜별 가능 여부 응답
- 최적 날짜 자동 추천
- 캘린더형 히트맵 시각화
- 외부 웹 UI 연동

## 5. 핵심 기능 정의

## 5-1. 세션 생성

관리자가 명령어로 세션을 생성한다.

예상 입력값:

- 세션명
- 세션 진행 날짜
- 응답 마감일시
- 공지 채널
- 참여 대상 역할

예시:

`/session create title:"금주 TRPG 세션" date:"2026-03-22 20:00" close:"2026-03-20 23:59" role:@TRPG`

생성 결과:

- DB에 세션 데이터 저장
- 공지 채널에 참여 체크 메시지 게시
- 버튼 3개 렌더링
  - 참석
  - 불참
  - 미정

---

## 5-2. 응답 수집

참여자는 버튼을 눌러 상태를 표시한다.

규칙:

- 1인 1상태만 유지
- 새 응답을 누르면 이전 상태를 덮어씀
- 응답 시 집계 숫자를 즉시 갱신할 수 있음

상태 값:

- YES
- NO
- MAYBE

---

## 5-3. 응답 대상자 기준

무응답자를 찾기 위해 기준 집단이 필요하다.

초기 기준 방식:

- 특정 Discord Role 기반

예:

- `@TRPG_MEMBER`

최종 집계 시:

- 역할 보유자 전체 목록 확보
- 응답 저장된 유저와 비교
- 무응답자 추출

---

## 5-4. 마감 처리

마감 시간이 되면:

- 해당 세션 상태를 CLOSED로 전환
- 버튼 비활성화
- 최종 집계 수행
- 결과 메시지 게시

최종 결과 예시:

- 세션명
- 진행 날짜
- 참석자 목록
- 불참자 목록
- 미정자 목록
- 무응답자 목록

---

## 5-5. 리마인드

확장 기능으로 추가한다.

예상 시점:

- 마감 24시간 전
- 마감 3시간 전

대상:

- 무응답자만 멘션

## 10. 캘린더형 확장 방향

캘린더형 결과는 MVP 완료 후 확장한다.

중요한 판단:

- 단일 날짜 세션에서는 캘린더가 핵심 기능이 아니라 보조 시각화임
- 캘린더가 진짜 의미 있으려면 다중 날짜 후보 조율 구조가 필요함

따라서 캘린더 확장 시 설계 변경 사항:

- 세션 1개에 후보 날짜 여러 개 저장
- 사용자 1명당 여러 날짜에 응답 가능
- 날짜별 가능 인원 수 집계
- 최적 날짜 추천 로직 추가
- 결과를 캘린더 히트맵 형태로 시각화

이 단계부터는 디스코드 내부 UI만으로는 한계가 있을 수 있으므로,
외부 웹 UI 연동을 고려한다.

## 부록 B. 현재 추천 결론

지금은 캘린더 중심 설계로 바로 가지 말고,
먼저 **단일 날짜 참석 체크형 MVP**를 완성하는 것이 맞다.

그 다음 실사용하면서 아래를 검증한다.

- 운영자들이 진짜 원하는 게 단일 일정 참석 체크인지
- 아니면 여러 날짜 중 최적 날짜 찾기인지
- 결과를 텍스트로 보는 게 충분한지
- 캘린더 시각화가 실제로 유용한지

이 검증이 끝난 뒤, 캘린더형 다중 날짜 구조로 확장한다.
